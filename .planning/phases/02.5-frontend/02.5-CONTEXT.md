# Phase 2.5: Frontend - Context

**Gathered:** 2026-04-30
**Status:** Ready for planning

<domain>
## Phase Boundary

Build the frontend application shell for PaperTrade — the UI that connects to the existing Phase 1 (data) and Phase 2 (trading/portfolio) backend APIs. This phase covers the full user-facing application: dashboard layout, portfolio view, order management, asset prices, and real-time data integration.

Charts, technical indicators, and drawing tools are explicitly Phase 3. External signals and AI advisor are Phases 4 and 5.

</domain>

<decisions>
## Implementation Decisions

### Project Setup & Tech Choices
- **D-01 (State management):** TanStack Query (React Query) for server state (orders, portfolio), paired with minimal Zustand store for real-time WebSocket price state. Industry standard React pattern.
- **D-02 (Routing):** React Router v6. Standard, well-documented, sufficient for 4-6 routes.
- **D-03 (API client):** Manual typed fetch wrappers per endpoint. No codegen — 12 endpoints is small enough. Can add OpenAPI codegen later if API grows.

### Layout & Navigation
- **D-04 (Navigation):** Left sidebar, collapsible. Standard trading dashboard pattern. Gives room to grow positions/orders/assets/settings.
- **D-05 (Mobile nav):** Bottom tab bar. Thumbs-friendly, persistent. 5 tabs max: Portfolio, Orders, Assets, Settings, plus active trade monitor.
- **D-06 (Color theme):** Dark mode only for v1. CSS variable architecture to allow light mode toggle later with minimal effort. Dark is industry standard for trading apps.

### Portfolio Page
- **D-07 (Layout):** Summary cards row (cash, total value, P&L, win rate) at top + full positions table below. Industry standard brokerage pattern (Zerodha, Robinhood, ETrade).
- **D-08 (Empty state):** Show cash balance prominently (100K INR), "No positions yet" message, "Place your first trade" CTA linking to orders page.

### Orders Page
- **D-09 (Order placement UX):** Side panel / drawer that slides from right. Keeps order list visible while placing. Standard trading UX.
- **D-10 (Order list):** Table with status badges (colored). Rows: symbol, side, qty, price, status, date, cancel action. Sortable columns. Data-dense standard view.

### Real-time & Data
- **D-11 (WS reconnection):** Auto-reconnect with exponential backoff (1s, 2s, 4s, 8s, cap 30s). "Reconnecting..." banner shown during disconnect. No manual retry needed.
- **D-12 (Stale price indicator):** Combined approach — configurable staleness threshold. Recent stale: dimmed text + tooltip ("Last updated: Xs ago"). Extended stale: yellow warning badge on price.

### Responsive & Polish
- **D-13 (Desktop density):** Moderate/balanced spacing. Comfortable row heights. Not as dense as Bloomberg terminals, not as airy as marketing sites.
- **D-14 (Error handling):** Inline error banners per section with retry button. Toasts dismiss too easily — user might miss "order failed." 404 pages show branded layout with nav links.

### Claude's Discretion
- Exact color palette and spacing values within dark theme
- Loading skeleton design
- Toast notification timing and placement
- Sidebar width and breakpoint for collapse
- Form validation styling details
- WebSocket hook implementation details

</decisions>

None — frontend starts from scratch. Backend API is fully built and tested (3804 lines Python, 11 passing tests).

### Established Patterns
- REST API at `/api/v1/*` with JSON responses
- WebSocket at `/ws/prices` broadcasting `{type: "prices", data: {SYMBOL: {price, currency, source, is_stale}}}` every 5s

- API docs at `/docs` (OpenAPI/Swagger)
- No auth headers needed (single-user)
- WebSocket auto-connects on page load
