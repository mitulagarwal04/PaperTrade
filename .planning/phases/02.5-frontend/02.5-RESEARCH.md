# Phase 2.5: Frontend Application - Research

**Researched:** 2026-04-30
**Domain:** React frontend application with real-time data integration
**Confidence:** HIGH

## Summary

This phase builds the frontend application shell for PaperTrade: a dark-mode trading dashboard with sidebar navigation, portfolio views, order management, real-time price display, and WebSocket integration. The frontend is a greenfield React application (no existing frontend code) connecting to the existing FastAPI backend at `http://localhost:8000`.

**Primary recommendation:** Scaffold with Vite 5 + React 18 + TypeScript 5, use shadcn/ui (Tailwind v3 mode) for component primitives, TanStack Query v5 for server state (portfolio/orders/asset data), Zustand v5 for WebSocket-driven real-time price state, and React Router v6 for routing. All locked decisions from CONTEXT.md are fully compatible with current library versions.

**Key tension identified:** shadcn/ui v4 defaults to Tailwind v4 + React 19, but supports Tailwind v3 + React 18 via its legacy path. The project's UI-SPEC.md and CLAUDE.md both specify React 18 and TailwindCSS 3.x. The init flow must pin to the v3-compatible path -- use `npx shadcn@latest init` in an existing Vite 5 + React 18 + Tailwind v3 project; shadcn's docs confirm "Your existing apps with Tailwind v3 and React 18 will still work" [CITED: ui.shadcn.com/docs/tailwind-v4].

## User Constraints (from CONTEXT.md)

### Locked Decisions

- **D-01 (State management):** TanStack Query (React Query) for server state (orders, portfolio), paired with minimal Zustand store for real-time WebSocket price state. Industry standard React pattern.
- **D-02 (Routing):** React Router v6. Standard, well-documented, sufficient for 4-6 routes.
- **D-03 (API client):** Manual typed fetch wrappers per endpoint. No codegen -- 12 endpoints is small enough. Can add OpenAPI codegen later if API grows.
- **D-04 (Navigation):** Left sidebar, collapsible. Standard trading dashboard pattern. Gives room to grow positions/orders/assets/settings.
- **D-05 (Mobile nav):** Bottom tab bar. Thumbs-friendly, persistent. 5 tabs max: Portfolio, Orders, Assets, Settings, plus active trade monitor.
- **D-06 (Color theme):** Dark mode only for v1. CSS variable architecture to allow light mode toggle later with minimal effort. Dark is industry standard for trading apps.
- **D-07 (Layout):** Summary cards row (cash, total value, P&L, win rate) at top + full positions table below. Industry standard brokerage pattern (Zerodha, Robinhood, ETrade).
- **D-08 (Empty state):** Show cash balance prominently (100K INR), "No positions yet" message, "Place your first trade" CTA linking to orders page.
- **D-09 (Order placement UX):** Side panel / drawer that slides from right. Keeps order list visible while placing. Standard trading UX.
- **D-10 (Order list):** Table with status badges (colored). Rows: symbol, side, qty, price, status, date, cancel action. Sortable columns. Data-dense standard view.
- **D-11 (WS reconnection):** Auto-reconnect with exponential backoff (1s, 2s, 4s, 8s, cap 30s). "Reconnecting..." banner shown during disconnect. No manual retry needed.
- **D-12 (Stale price indicator):** Combined approach -- configurable staleness threshold. Recent stale: dimmed text + tooltip ("Last updated: Xs ago"). Extended stale: yellow warning badge on price.
- **D-13 (Desktop density):** Moderate/balanced spacing. Comfortable row heights. Not as dense as Bloomberg terminals, not as airy as marketing sites.
- **D-14 (Error handling):** Inline error banners per section with retry button. Toasts dismiss too easily -- user might miss "order failed." 404 pages show branded layout with nav links.

- Loading skeleton design

- WebSocket hook implementation details

### Pattern 1: TanStack Query v5 Custom Hooks (Server State)

**What:** Wrap each API endpoint group in a custom hook that uses `useQuery` or `useMutation`. Each hook encapsulates the query key, fetch function, cache configuration, and error handling.

**When to use:** Every API call that reads server data (portfolio, positions, orders, assets).

**Example:**
```typescript
// src/hooks/usePortfolio.ts
import { useQuery } from "@tanstack/react-query";
import { api } from "@/lib/api";

interface PortfolioSummary {
  cash_inr: number;
  total_value_inr: number;
  unrealized_pnl_inr: number;
  win_rate: number;
  positions: Position[];
}

export function usePortfolio() {
  return useQuery<PortfolioSummary>({
    queryKey: ["portfolio"],
    queryFn: () => api.get("/api/v1/portfolio"),
    refetchInterval: 10_000, // Poll every 10s as fallback
    staleTime: 5_000,
  });
}
```

### Pattern 2: useMutation for Order Operations

**What:** Use `useMutation` for create/delete operations (place order, cancel order). Handle loading, error, and success states. Invalidate related queries on success.

**When to use:** POST /api/v1/orders (place order), DELETE /api/v1/orders/{id} (cancel order), POST /api/v1/portfolio/reset (reset portfolio).

**Example:**
```typescript
// src/hooks/useOrderMutations.ts
import { useMutation, useQueryClient } from "@tanstack/react-query";
import { api } from "@/lib/api";

interface PlaceOrderParams {
  symbol: string;
  side: "BUY" | "SELL";
  order_type: "MARKET" | "LIMIT" | "STOP_LOSS" | "TAKE_PROFIT";
  quantity: number;
  price?: number;
  stop_price?: number;
}

export function usePlaceOrder() {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: (params: PlaceOrderParams) =>
      api.post("/api/v1/orders", params),
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: ["orders"] });
      queryClient.invalidateQueries({ queryKey: ["portfolio"] });
    },
  });
}

export function useCancelOrder() {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: (orderId: number) =>
      api.delete(`/api/v1/orders/${orderId}`),
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: ["orders"] });
      queryClient.invalidateQueries({ queryKey: ["portfolio"] });
    },
  });
}
```

### Pattern 3: Zustand v5 Price Store (WebSocket State)

**What:** Minimal Zustand store for real-time price data from WebSocket. Single store with prices map, connection status, and update methods.

**When to use:** All WebSocket price state. Separated from TanStack Query because prices are pushed, not pulled.

**Example:**
```typescript
// src/stores/priceStore.ts
import { create } from "zustand";

interface PriceData {
  price: number;
  currency: string;
  source: string;
  is_stale?: boolean;
  last_updated: number;
}

interface PriceStore {
  prices: Record<string, PriceData>;
  wsStatus: "connecting" | "connected" | "disconnected" | "reconnecting";
  reconnectAttempt: number;
  updatePrices: (data: Record<string, Omit<PriceData, "last_updated">>) => void;
  setWsStatus: (status: PriceStore["wsStatus"]) => void;
  incrementReconnect: () => void;
  resetReconnect: () => void;
}

export const usePriceStore = create<PriceStore>((set) => ({
  prices: {},
  wsStatus: "disconnected",
  reconnectAttempt: 0,
  updatePrices: (data) =>
    set((state) => {
      const now = Date.now();
      const updated: Record<string, PriceData> = {};
      for (const [symbol, price] of Object.entries(data)) {
        updated[symbol] = {
          ...price,
          last_updated: now,
        };
      }
      return {
        prices: { ...state.prices, ...updated },
      };
    }),
  setWsStatus: (status) => set({ wsStatus: status }),
  incrementReconnect: () =>
    set((state) => ({ reconnectAttempt: state.reconnectAttempt + 1 })),
  resetReconnect: () => set({ reconnectAttempt: 0 }),
}));
```

### Pattern 4: WebSocket Custom Hook with Exponential Backoff

**What:** Custom React hook that manages a WebSocket connection with automatic reconnection using exponential backoff (1s, 2s, 4s, 8s, cap 30s). Updates Zustand store on message receipt.

**When to use:** Exactly once in the app, typically at the App.tsx level or in a provider.

**Example:**
```typescript
// src/hooks/useWebSocket.ts
import { useEffect, useRef } from "react";
import { usePriceStore } from "@/stores/priceStore";
import { WS_URL, MAX_BACKOFF } from "@/lib/constants";

export function useWebSocket() {
  const wsRef = useRef<WebSocket | null>(null);
  const timerRef = useRef<ReturnType<typeof setTimeout> | null>(null);
  const {
    setWsStatus,
    updatePrices,
    reconnectAttempt,
    incrementReconnect,
    resetReconnect,
  } = usePriceStore();

  const connect = () => {
    setWsStatus("connecting");
    const ws = new WebSocket(WS_URL);
    wsRef.current = ws;

    ws.onopen = () => {
      setWsStatus("connected");
      resetReconnect();
    };

    ws.onmessage = (event) => {
      try {
        const message = JSON.parse(event.data);
        if (message.type === "prices") {
          updatePrices(message.data);
        }
      } catch {
        // Ignore malformed messages
      }
    };

    ws.onclose = () => {
      setWsStatus("reconnecting");
      scheduleReconnect();
    };

    ws.onerror = () => {
      ws.close();
    };
  };

  const scheduleReconnect = () => {
    const delay = Math.min(1000 * Math.pow(2, reconnectAttempt), MAX_BACKOFF);
    timerRef.current = setTimeout(() => {
      incrementReconnect();
      connect();
    }, delay);
  };

  useEffect(() => {
    connect();
    return () => {
      if (wsRef.current) {
        wsRef.current.close();
      }
      if (timerRef.current) {
        clearTimeout(timerRef.current);
      }
    };
  }, []);
}
```

### Pattern 5: App Layout with React Router v6 createBrowserRouter

**What:** Use React Router v6 data router pattern with `createBrowserRouter` and `RouterProvider`. Define routes with parent layout + child pages. Use `<Outlet />` in the layout for child rendering.

**When to use:** Always. This is the recommended v6 pattern for new apps.

**Example:**
```typescript
// src/main.tsx

const router = createBrowserRouter([
  {
    path: "/",
    element: <App />,
    errorElement: <NotFoundPage />,
    children: [
      { index: true, element: <DashboardPage /> },
      { path: "portfolio", element: <PortfolioPage /> },
      { path: "orders", element: <OrdersPage /> },
      { path: "assets", element: <AssetsPage /> },
      { path: "settings", element: <SettingsPage /> },
    ],
  },

createRoot(document.getElementById("root")!).render(
  <QueryClientProvider client={queryClient}>
```

### Pattern 6: TanStack Query + Zustand Integration (Stale Price Display)

**What:** Combine TanStack Query portfolio data (which contains position current_price) with Zustand WebSocket prices to show live-updating values. The portfolio response includes `current_price` and `current_price_inr` from the last API call; the WS price store provides more recent values between refetches.

**When to use:** Price-sensitive displays where live data improves UX but API data is authoritative.

**Example:**
```typescript
// Inside PositionsTable.tsx
const { data: portfolio } = usePortfolio();
const livePrices = usePriceStore((state) => state.prices);

// Enrich positions with live prices from WS
const enrichedPositions = portfolio?.positions.map((pos) => {
  const live = livePrices[pos.symbol];
  const currentPrice = live?.price ?? pos.current_price;
  const isStale = live ? isPriceStale(live.last_updated) : false;
  // Recalculate market value with live price
  const marketValue = Number(pos.quantity) * Number(currentPrice);
  const unrealizedPnl = marketValue - Number(pos.total_cost_inr);

  return { ...pos, displayPrice: currentPrice, displayPnl: unrealizedPnl, isStale };
});
```

### Anti-Patterns to Avoid

- **Putting WebSocket logic in individual page components:** The WS connection should be initiated once at the app root level (in App.tsx or a provider), not per-page. Pages consume from the Zustand store.
- **TanStack Query `refetchInterval` as primary real-time mechanism:** Use WebSocket for real-time prices, not polling. TanStack Query polling is the fallback for portfolio/orders data that changes less frequently.
- **Mixing TanStack Query and Zustand for the same data:** Server data (portfolio, orders) goes in TanStack Query. WebSocket data (prices) goes in Zustand. They do not overlap.
- **Direct WebSocket state management in components:** Never create `new WebSocket()` in a component. Always use the custom hook.
- **Mutable Zustand store updates:** Always use `set()` in Zustand with new object references, never mutate `state.prices[symbol]` directly.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| UI component primitives | Custom buttons, modals, tooltips, dropdowns | shadcn/ui (Radix primitives) | Accessibility, keyboard nav, focus management, ARIA attributes. Radix handles all edge cases |
| Server state caching | Custom fetch + cache layer | TanStack Query v5 | Deduplication, background refetch, cache invalidation, loading/error states, stale-while-revalidate |
| Real-time client state | Custom event emitter | Zustand v5 | Minimal boilerplate, selector-based re-renders, no providers, DevTools support |
| Routing | Window.location + history API | React Router v6 | URL parsing, navigation lifecycle, lazy loading, scroll restoration, nested routes |
| CSS reset and base | Custom CSS reset | TailwindCSS preflight | Consistent cross-browser defaults, proven base |
| Class name merging | String concatenation | clsx + tailwind-merge (cn utility) | Proper handling of conditional classes, Tailwind class conflicts |
| Icons | Custom SVG components | lucide-react | 1000+ consistent icons, tree-shakeable, React components with standard props |
| Date formatting | Manual date strings | Intl.DateTimeFormat (built-in) | Locale-aware, no library needed. Use native Intl API for all date/number formatting |

**Key insight:** Trading dashboards have complex UI requirements (accessible controls, responsive data tables, real-time updates) but the complexity is in integration, not in building primitives from scratch. Every hand-rolled UI primitive introduces accessibility bugs, keyboard navigation gaps, and maintenance burden. shadcn/ui + TanStack Query + Zustand cover 90% of the plumbing, leaving the project to focus on trading-specific business logic.

## Common Pitfalls

### Pitfall 1: Vite 5 Proxy Not Configured for Backend API
**What goes wrong:** API calls fail with CORS errors or `http://localhost:5173/api/v1/portfolio` doesn't resolve.
**Why it happens:** Vite dev server runs on port 5173, backend on 8000. Without proxy config, the browser makes CORS requests that fail if the backend isn't configured (it is configured with `allow_origins=["*"]`, but proxy is still cleaner).
**How to avoid:** Configure proxy in `vite.config.ts`:
```typescript
export default defineConfig({
  plugins: [react()],
  server: {
    proxy: {
      "/api": "http://localhost:8000",
      "/ws": {
        target: "ws://localhost:8000",
        ws: true,
      },
    },
  },
});
```
**Warning signs:** Browser console shows CORS errors or `ERR_CONNECTION_REFUSED` on API calls. Alternatively, use CORS directly (already configured on backend) by hitting `http://localhost:8000` directly -- but proxy is simpler for dev.

### Pitfall 2: TanStack Query v5 - Using `isLoading` Instead of `isPending`
**What goes wrong:** TypeScript errors or stale loading states because `isLoading` was removed.
**Why it happens:** TanStack Query v5 renamed `isLoading` to `isPending` for `useQuery`. The old `isLoading` only existed in v4 and is removed.
**How to avoid:** Always destructure `isPending`, `isFetching`, `isError`, `data`, `error` from `useQuery()` results. Never use `isLoading`.
**Warning signs:** TypeScript compilation errors on `isLoading`.

### Pitfall 3: Zustand v5 - `create<State>()()` Double Parentheses
**What goes wrong:** TypeScript errors when creating a store.
**Why it happens:** Zustand v5 requires the `create<State>()(fn)` pattern with double parentheses for TypeScript type inference.
**How to avoid:**
```typescript
// Correct v5 pattern
interface MyStore { ... }
export const useStore = create<MyStore>()((set) => ({ ... }));
// Not: create<MyStore>((set) => ({}))
```

### Pitfall 4: React Router v6 - Using `<BrowserRouter>` Instead of `createBrowserRouter`
**What goes wrong:** Missing data API features (loaders, actions), and non-standard pattern.
**Why it happens:** Both `<BrowserRouter>` (legacy) and `createBrowserRouter` (v6.4+) are exported. Beginners use the simpler-looking `<BrowserRouter>`.
**How to avoid:** Always use `createBrowserRouter` + `RouterProvider`. This is the recommended v6 pattern and provides access to loaders, actions, and error boundaries at the route level.

### Pitfall 5: shadcn/ui Tailwind v3 Setup with Latest CLI
**What goes wrong:** `npx shadcn@latest init` may prompt for or default to Tailwind v4/React 19 configuration.
**Why it happens:** shadcn/ui v4+ defaults to Tailwind v4 for new projects.
**How to avoid:** Set up Tailwind v3 first (with `tailwind.config.js`, `postcss.config.js`, `@tailwind` directives), then run `npx shadcn@latest init`. The CLI detects the v3 setup and configures compatibility mode. shadcn explicitly documents that "Your existing apps with Tailwind v3 and React 18 will still work" [CITED: ui.shadcn.com/docs/tailwind-v4].

### Pitfall 6: WebSocket Cleanup on Unmount
**What goes wrong:** Memory leaks, stale connections, or "reconnecting..." spam when navigating between pages.
**Why it happens:** If the WebSocket is created in a component that mounts/unmounts (like a page component), each mount creates a new connection.
**How to avoid:** Initialize WebSocket once at the App root level (App.tsx), not in individual pages. The `useEffect` cleanup in the custom hook must close the WebSocket and clear timers.

### Pitfall 7: Dark Mode Flicker on Page Load
**What goes wrong:** Brief flash of light background before dark mode applies.
**Why it happens:** Tailwind dark mode via class strategy requires the `dark` class to be set before React hydrates. If the class is set in a component effect, the initial render uses light mode.
**How to avoid:** Set the `dark` class in `index.html` `<html>` tag or via an inline script before React loads:
```html
<script>
  document.documentElement.classList.add('dark');
</script>
```

### API Client Wrapper (Manual Typed Fetch)

```typescript
// src/lib/api.ts
const BASE_URL = ""; // Empty when using Vite proxy

class ApiError extends Error {
  constructor(public status: number, message: string) {
    super(message);
    this.name = "ApiError";
  }
}

async function request<T>(path: string, options?: RequestInit): Promise<T> {
  const res = await fetch(`${BASE_URL}${path}`, {
    headers: {
      "Content-Type": "application/json",
    },
    ...options,
  });

  if (!res.ok) {
    const body = await res.json().catch(() => ({}));
    throw new ApiError(res.status, body.detail || res.statusText);
  }

  return res.json();
}

export const api = {
  get: <T>(path: string) => request<T>(path),
  post: <T>(path: string, body: unknown) =>
    request<T>(path, {
      method: "POST",
      body: JSON.stringify(body),
    }),
  delete: <T>(path: string, body?: unknown) =>
    request<T>(path, {
      method: "DELETE",
      body: body ? JSON.stringify(body) : undefined,
    }),
};
```

```typescript
export const EXTENDED_STALE_THRESHOLD_MS = 30_000;
export const WS_URL = `${location.protocol === "https:" ? "wss:" : "ws:"}//${location.host}/ws/prices`;

// src/lib/utils.ts
import { clsx, type ClassValue } from "clsx";
import { twMerge } from "tailwind-merge";

export function cn(...inputs: ClassValue[]) {
  return twMerge(clsx(inputs));
}

export function isPriceStale(lastUpdated: number): "fresh" | "stale" | "extended" {
  const age = Date.now() - lastUpdated;
  if (age < STALE_THRESHOLD_MS) return "fresh";
  if (age < EXTENDED_STALE_THRESHOLD_MS) return "stale";
  return "extended";
}
```
