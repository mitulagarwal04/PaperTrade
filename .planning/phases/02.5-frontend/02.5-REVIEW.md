# Phase 02.5: Code Review Report

**Depth:** standard

## Summary

The frontend codebase shows solid React/TypeScript conventions, proper Tailwind usage, well-structured zustand stores, and good accessibility patterns (aria-labels, role attributes, focus-visible rings). However, there is one critical dependency management issue that will cause CI/CD build failures, several runtime bugs (stale closures, missing toast renderer, currency mismatch in position calculations), and a handful of code quality items.

The most pressing issue is that 98 packages (all Radix UI primitives, class-variance-authority, floating-ui) are present in `node_modules` as extraneous -- not declared in `package.json`. Running `npm ci` will produce a broken build. This must be fixed before any CI/CD pipeline is attached.

---

## Critical Issues

### CR-01: Extraneous runtime dependencies break `npm ci` builds

**File:** `frontend/package.json` (all)
**Issue:** The shadcn/ui components (`button.tsx`, `dialog.tsx`, `select.tsx`, `sheet.tsx`, `toast.tsx`, etc.) depend on several runtime packages that are NOT declared in `package.json`. All 98 are flagged as `extraneous` by `npm ls`. These include:

- `@radix-ui/react-dialog` (used by `dialog.tsx`, `sheet.tsx`)
- `@radix-ui/react-select` (used by `select.tsx`)
- `@radix-ui/react-toast` (used by `toast.tsx`)
- `@radix-ui/react-slot` (used by `button.tsx`)
- `@radix-ui/react-dropdown-menu` (used by `dropdown-menu.tsx`)
- `@radix-ui/react-tabs`, `@radix-ui/react-tooltip`, `@radix-ui/react-separator`
- `class-variance-authority` (used by `button.tsx`, `toast.tsx`, `sheet.tsx`)
- `@floating-ui/*` (transitive dep of radix packages)

Since these are not in `package.json`, they are also absent from `package-lock.json`. Running `npm ci` (which deletes `node_modules` and reinstalls from the lockfile) will result in a broken build -- all UI component imports will fail at runtime.

**Fix:** Add all direct runtime dependencies to `package.json`:

```json
{
  "dependencies": {
    "@radix-ui/react-dialog": "^1.1.6",
    "@radix-ui/react-dropdown-menu": "^2.1.6",
    "@radix-ui/react-select": "^2.1.6",
    "@radix-ui/react-separator": "^1.1.2",
    "@radix-ui/react-slot": "^1.1.2",
    "@radix-ui/react-tabs": "^1.1.3",
    "@radix-ui/react-toast": "^1.2.6",
    "@radix-ui/react-tooltip": "^1.1.8",
    "class-variance-authority": "^0.7.1"
  }
}
```

Then run `npm install` to update `package-lock.json`.

## Warnings

### WR-01: WebSocket reconnection exponential backoff non-functional due to stale closure

**File:** `frontend/src/hooks/useWebSocket.ts:16-53`

**Issue:** The `connect()` and `scheduleReconnect()` functions are defined as plain closures inside the hook body. They capture `reconnectAttempt` as a primitive value from the initial render (always `0`). The `useEffect` with `[]` deps captures these first-render closures. When the WebSocket disconnects and reconnects, every reconnection delay is computed as `Math.min(1000 * Math.pow(2, 0), 30000)` = 1000ms. The `incrementReconnect()` calls DO update the store state, but the scheduled callback always reads the stale captured `reconnectAttempt = 0`.

This means the client will hammer the server every 1 second during an outage, never backing off.

**Fix:** Read `reconnectAttempt` from the zustand store directly inside `scheduleReconnect` using `usePriceStore.getState()`:

```ts
const scheduleReconnect = () => {
  const attempt = usePriceStore.getState().reconnectAttempt;
  const delay = Math.min(1000 * Math.pow(2, attempt), MAX_BACKOFF);
  timerRef.current = setTimeout(() => {
    usePriceStore.getState().incrementReconnect();
    connect();
  }, delay);
};
```

This avoids the stale closure because `getState()` always returns the current store value.

### WR-02: Toast notifications never rendered to user

**File:** `frontend/src/components/orders/OrderForm.tsx:69` calls `toast()` on successful order placement.
**File:** `frontend/src/App.tsx` and `frontend/src/main.tsx` -- neither renders `<Toaster />`.

**Issue:** The `OrderForm` calls the `toast()` function from `@/hooks/use-toast` when an order is placed successfully. This dispatches a toast to the global state. However, the `<Toaster />` component (from `@/components/ui/toaster`) is never mounted in the component tree. Without `<Toaster />`, the `useToast()` subscribers never get a rendering surface, so toast notifications are invisible.

**Fix:** Add `<Toaster />` to the app layout, typically in `App.tsx`:

```tsx
import { Toaster } from "@/components/ui/toaster";

export default function App() {
  useWebSocket();
  return (
    <>
      <div className="flex h-screen bg-background text-primary">
        {/* existing layout */}
      </div>
      <Toaster />
    </>
  );
}
```

### WR-03: Currency mismatch in PositionsTable enrichment

**File:** `frontend/src/components/portfolio/PositionsTable.tsx:27-33`

**Issue:** When enriching positions with live WebSocket prices, the code computes:

```ts
const currentPrice = live?.price ?? pos.current_price;
const marketValue = Number(pos.quantity) * Number(currentPrice);
const unrealizedPnl = marketValue - Number(pos.total_cost_inr);
```

Here, `currentPrice` is in the asset's native currency (e.g., USD for AAPL), while `total_cost_inr` is in INR. Multiplying quantity by a non-INR price gives a market value in the wrong currency, and subtracting INR-cost from it produces a meaningless P&L figure.

The API provides `current_price_inr` and `market_value_inr` on the `Position` interface. The WebSocket store's `price` is also in native currency. The enrichment should either use the API's pre-converted INR values or convert the WS price using the stored `currency` field.

**Fix:** Use `pos.current_price_inr` for the fallback and convert the WS price:

```ts
const enrichedPositions = useMemo(() => {
  if (!positions) return [];
  return positions.map((pos) => {
    const live = livePrices[pos.symbol];
    const currentPriceInr = live
      ? convertToInr(live.price, pos.currency) // requires FX rate lookup
      : Number(pos.current_price_inr);
    const marketValue = Number(pos.quantity) * currentPriceInr;
    const unrealizedPnl = marketValue - Number(pos.total_cost_inr);
    // ...
  });
}, [positions, livePrices]);
```

If FX rates are not yet available in the frontend store, a safer interim fix is to use the API-provided INR values and only overlay the stale/live indicator from the WS data without re-computing P&L:

```ts
const currentPrice = live?.price ?? pos.current_price;
const staleness = live ? isPriceStale(live.last_updated) : "fresh";
// Use API values for market_value_inr and unrealized_pnl_inr directly
return {
  ...pos,
  displayPrice: currentPrice,
  displayMarketValue: Number(pos.market_value_inr),
  displayPnl: Number(pos.unrealized_pnl_inr),
  pnlPercent: ...,
  isStale: staleness !== "fresh",
};
```

### WR-04: Type escape via `params as any` in OrderForm

**File:** `frontend/src/components/orders/OrderForm.tsx:67`

**Issue:** The `handleSubmit` function builds a `params: Record<string, unknown>` object and passes it to `placeOrder.mutate(params as any, ...)`. The `as any` cast bypasses TypeScript's type check for the `PlaceOrderParams` interface. This means a malformed payload (e.g., `order_type: "MARKET"` with an accidental `price` field) will be sent to the API without compile-time validation.

**Fix:** Type the `params` object as `PlaceOrderParams` directly instead of `Record<string, unknown>`:

```ts
const params: PlaceOrderParams = {
  symbol: sym,
  side,
  order_type: orderType as PlaceOrderParams["order_type"],
  quantity: qty,
};
if (orderType === "LIMIT" && price) params.price = parseFloat(price);
if (["STOP_LOSS", "TAKE_PROFIT"].includes(orderType) && stopPrice) params.stop_price = parseFloat(stopPrice);
```

Then pass `params` directly without the type cast.

### IN-02: Staleness age indicator does not refresh independently

**File:** `frontend/src/components/shared/StalePriceBadge.tsx:17-18`
**File:** `frontend/src/components/assets/AssetPriceRow.tsx:27-28`

**Issue:** Both components compute `ageInSeconds = Math.floor((Date.now() - priceData.last_updated) / 1000)` at render time. If no new price data arrives, the component will not re-render, and the displayed "Last updated: Ns ago" value becomes increasingly inaccurate over time. The age indicator freezes until a new price update triggers a re-render.

**Fix (optional):** For a more accurate age display, add a `useEffect` with an interval timer that force-re-renders the component periodically (e.g., every 5 seconds) while a stale price exists. Alternatively, accept this as a minor UX limitation since the tooltip is informational and the staleness badge (extended) remains correct.

### IN-03: SummaryCards `win_rate` display assumes decimal value

**File:** `frontend/src/components/portfolio/SummaryCards.tsx:54`

**Issue:** Win rate is displayed as `(Number(card.value) * 100).toFixed(1) + "%"`, which multiplies by 100. This assumes the backend returns win_rate as a decimal (e.g., 0.75 for 75%). If the backend returns it as a percentage (e.g., 75 for 75%), this will display 7500%.

**Fix:** Verify the backend contract for `PortfolioSummary.win_rate`. If it is a decimal (0-1), the current code is correct. If it is a percentage (0-100), remove the `* 100` multiplication. Document the expected contract in the `PortfolioSummary` interface.

### IN-04: TableSkeleton uses non-deterministic `Math.random()` for column widths

**File:** `frontend/src/components/shared/LoadingSkeleton.tsx:45`

**Issue:** The `TableSkeleton` component generates random column widths on every render:

```tsx
style={{ width: `${60 + Math.random() * 40}px` }}
```

This causes non-deterministic renders -- each time the component renders, the widths differ. This produces inconsistent loading states between renders and causes unnecessary style recalculations.

**Fix:** Use deterministic widths. A simple approach is to distribute widths based on column index:

```tsx
const widths = [80, 60, 100, 70, 90];
// ...
style={{ width: `${widths[j % widths.length]}px` }}
```

### IN-05: SettingsPage dialog does not close after successful portfolio reset

**File:** `frontend/src/pages/SettingsPage.tsx:22-36`

**Issue:** After `handleReset` succeeds, `setResetDone(true)` is called but `setShowReset(false)` is NOT called. The confirmation dialog remains open. The user must manually dismiss it (click outside or press Escape) before seeing the success message.

**Fix:** Close the dialog after successful reset:

```tsx
const handleReset = async () => {
  setIsResetting(true);
  setResetError(null);
  try {
    await api.post("/api/v1/portfolio/reset", { confirm: true });
    setResetDone(true);
    setShowReset(false);  // Close the dialog
    queryClient.invalidateQueries({ queryKey: ["portfolio"] });
    queryClient.invalidateQueries({ queryKey: ["positions"] });
    queryClient.invalidateQueries({ queryKey: ["orders"] });
  } catch (err) {
    setResetError((err as Error)?.message || "Reset failed");
  } finally {
    setIsResetting(false);
  }
};
```

### IN-06: OrderForm reference price lookup missing `.trim()`

**File:** `frontend/src/components/orders/OrderForm.tsx:38` and `:44`

**Issue:** The reference price lookup on line 38 uses `symbol.toUpperCase()`, while the validation on line 44 uses `symbol.toUpperCase().trim()`. If the user types "AAPL " (with trailing space), the reference price lookup will fail to find `"AAPL "` in the store (the store keys have no spaces), but the order will still be submitted with the trimmed symbol. Minor inconsistency.

```ts
// Line 38: no .trim()
const referencePrice = usePriceStore((state) =>
  symbol ? state.prices[symbol.toUpperCase()]?.price : undefined
);
// Line 44: .trim() present
const sym = symbol.toUpperCase().trim();
```

**Fix:** Add `.trim()` to the reference price lookup:

```ts
const referencePrice = usePriceStore((state) =>
  symbol ? state.prices[symbol.toUpperCase().trim()]?.price : undefined
);
```
